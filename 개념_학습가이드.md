## 1. 이 과제는 무엇을 배우는가?

### 1.1 한 줄 요약

> **"실서버에서 장애가 터졌을 때, 재부팅 대신 원인을 찾아 기록하는 능력"**

### 1.2 과제의 핵심 3가지

| 장애 유형 | 한 줄 정의 | 증상 |
|---|---|---|
| **Memory Leak / OOM** | 메모리가 조금씩 새어나가 결국 프로세스가 죽는 현상 | 프로세스가 갑자기 종료됨 |
| **CPU 과점유** | 특정 프로세스가 CPU를 독점하여 시스템 전체가 느려지는 현상 | 응답 지연, Watchdog에 의한 종료 |
| **Deadlock** | 두 스레드가 서로의 자원을 영원히 기다리며 멈추는 현상 | 프로세스가 살아있지만 아무것도 안 함 |

### 1.3 최종 목표

이 과제를 마치면 다음을 스스로 할 수 있게 된다.

- 로그와 관제 데이터를 읽고 **장애의 원인을 추론**할 수 있다
- **환경변수를 조정**하여 장애 재현 및 완화를 실험할 수 있다
- **GitHub Issue** 형식으로 동료에게 장애를 명확하게 전달할 수 있다

## 2. 왜 이 내용이 필요한가?

### 2.1 현업에서 벌어지는 일

개발자가 코드를 작성하고 배포하면, 그 코드는 **실서버(운영 환경)** 에서 24시간 돌아간다. 여기서 문제가 생기면 다음과 같은 상황이 펼쳐진다.

```
오전 3시 — 서비스 응답 없음
온콜 개발자 호출
"일단 재부팅합시다" → 재부팅 → 서비스 복구
→ 다음 날 밤 또 같은 현상
→ 그 다음 날 밤 또...
```

**재부팅은 증거를 없앤다.** 원인을 모르면 같은 장애가 반복된다.

### 2.2 역사적 배경

| 연도 | 사건 |
|---|---|
| **1960년대** | 초기 OS 연구에서 Deadlock 개념 공식화 (Dijkstra, 1965) |
| **1969년** | UNIX 탄생 — 프로세스, 파일, 파이프 추상화 도입 |
| **1970년대** | 가상 메모리(Virtual Memory) 개념 등장 — 메모리 관리 복잡도 증가 |
| **1980~90년대** | C/C++ 시대 — 개발자가 직접 메모리 할당/해제 → Memory Leak 빈번 발생 |
| **2000년대** | Java, Python 등 GC(Garbage Collector) 언어 등장 — 자동 메모리 관리, 하지만 GC도 완벽하지 않음 |
| **2010년대** | 클라우드·마이크로서비스 시대 — 수백 개 프로세스가 동시에 실행, 장애 분석이 더 복잡해짐 |
| **현재** | Prometheus, Grafana 등 관제 도구가 표준화, 로그 기반 트러블슈팅이 핵심 역량 |

### 2.3 이 과제가 주는 것

- 단순 코딩이 아닌 **운영(Operation)** 관점의 시야
- 추상적인 OS 개념을 **직접 눈으로 확인**하는 경험
- 현업에서 통하는 **기술 문서 작성** 능력

## 3. 기반 지식: 운영체제와 프로세스

### 3.1 운영체제(OS)란?

운영체제는 하드웨어(CPU, 메모리, 디스크)와 소프트웨어(프로그램) 사이의 **중재자**다.

```
┌──────────────────────────────┐
│     사용자 프로그램 (App)      │
├──────────────────────────────┤
│     운영체제 (OS Kernel)       │  ← CPU, 메모리, 파일, 프로세스 관리
├──────────────────────────────┤
│     하드웨어 (CPU / RAM)       │
└──────────────────────────────┘
```

OS의 핵심 역할:
- **프로세스 관리**: 어떤 프로그램이 CPU를 쓸지 결정
- **메모리 관리**: 프로그램에게 메모리를 할당하고 회수
- **파일 시스템**: 디스크의 데이터를 파일로 관리

### 3.2 프로세스(Process)란?

**프로세스 = 실행 중인 프로그램**

프로그램은 디스크에 있는 정적인 파일이다. 이것을 실행하면 OS가 메모리를 할당하고, CPU 시간을 주어 살아움직이게 만드는데, 이 살아있는 상태를 **프로세스**라고 한다.

```
disk의 agent-leak-app (파일)
        ↓  실행(./agent-leak-app)
메모리에 올라간 프로세스 (PID: 12345)
        ↓  CPU가 코드를 실행
출력, 네트워크 응답, 로그 기록...
```

#### 프로세스 상태

```
생성(Created)
    ↓
준비(Ready) ──→ 실행(Running) ──→ 대기(Waiting)
    ↑                ↓                  ↓
    └────────────────┘──────────────────┘
                    ↓
              종료(Terminated)
```

| 상태 | 설명 |
|---|---|
| Ready | CPU를 기다리는 상태 |
| Running | CPU를 점유하여 실행 중 |
| Waiting | I/O나 이벤트를 기다리는 상태 (CPU 미사용) |
| Terminated | 종료됨 |

#### Deadlock과의 연결

Deadlock은 프로세스(또는 스레드)가 **Waiting 상태에서 영원히 빠져나오지 못하는** 상황이다.

### 3.3 스레드(Thread)란?

- **프로세스**: 독립적인 실행 단위 (메모리 공간 분리)
- **스레드**: 프로세스 안의 더 작은 실행 단위 (메모리 공간 공유)

```
프로세스 (agent-leak-app)
├── Thread-A  → Task 처리
├── Thread-B  → Task 처리
└── Thread-C  → 모니터링
     (세 스레드가 같은 메모리 공간을 공유)
```

스레드가 메모리를 공유하기 때문에 **자원 경쟁(Race Condition)** 과 **Deadlock** 이 발생할 수 있다.

## 4. 메모리 구조 완전 이해

### 4.1 메모리의 종류

| 종류 | 속도 | 용도 |
|---|---|---|
| CPU 레지스터 | 가장 빠름 | 현재 연산 중인 값 |
| 캐시(Cache) | 매우 빠름 | 자주 쓰는 데이터 임시 저장 |
| RAM (주 메모리) | 빠름 | 실행 중인 프로그램 데이터 |
| 디스크 (SSD/HDD) | 느림 | 파일, 영구 저장 |

**Memory Leak은 RAM 영역에서 발생한다.**

### 4.2 프로세스 메모리 구조

프로세스가 실행될 때 OS는 아래와 같은 메모리 레이아웃을 만들어준다.

```
높은 주소
┌──────────────────┐
│      Stack       │  ← 함수 호출 정보, 지역 변수 (자동 관리)
│        ↓         │
│                  │
│        ↑         │
│       Heap       │  ← 동적 할당 메모리 (개발자/GC가 관리)
├──────────────────┤
│   BSS Segment    │  ← 초기화되지 않은 전역 변수
├──────────────────┤
│  Data Segment    │  ← 초기화된 전역 변수
├──────────────────┤
│  Text Segment    │  ← 실행 코드 (프로그램 자체)
└──────────────────┘
낮은 주소
```

#### Stack vs Heap

| | Stack | Heap |
|---|---|---|
| 관리 주체 | OS가 자동 | 개발자 or GC |
| 크기 | 작고 고정 | 크고 동적 |
| 속도 | 빠름 | 상대적으로 느림 |
| Memory Leak 위험 | 없음 | **있음** |

**Memory Leak은 Heap에서 할당한 메모리를 해제하지 않을 때 발생한다.**

## 5. Memory Leak & OOM

### 5.1 메모리 누수(Memory Leak)란?

프로그램이 Heap 메모리를 할당한 뒤, **다 사용했음에도 해제하지 않는** 현상.

#### C 언어 예시 (직접 메모리 관리)

```c
void process_request() {
    char *buffer = malloc(1024);  // 1KB 할당
    // 작업 수행...
    // free(buffer);  ← 이 줄이 빠지면 Memory Leak!
}
```

이 함수가 1초에 100번 호출되면, 1초마다 100KB씩 메모리가 쌓인다.

#### Python/Java 예시 (GC 언어에서도 발생)

```python
cache = []

def process():
    data = load_large_data()  # 수 MB짜리 데이터
    cache.append(data)        # 캐시에 추가
    # 하지만 cache는 절대 비워지지 않음
    # → GC가 있어도 참조가 남아있으면 해제 불가
```

**결론: GC 언어도 참조(reference)가 남아있으면 메모리를 해제할 수 없다.**

### 5.2 메모리 누수가 시스템에 미치는 영향

```
메모리 누수 시작
    ↓
Heap 점점 증가 (5% → 30% → 60% → 90%...)
    ↓
사용 가능한 메모리 감소
    ↓
OS가 스왑(Swap) 영역 사용 시작 (디스크를 RAM처럼 씀 → 매우 느려짐)
    ↓
메모리 고갈
    ↓
OOM(Out of Memory) 발생 → 프로세스 강제 종료
```

### 5.3 OOM (Out of Memory)

시스템 메모리가 완전히 고갈되면 OS 또는 애플리케이션의 보호 정책이 작동한다.

#### Linux OOM Killer

Linux 커널에는 **OOM Killer** 가 내장되어 있다. 메모리가 부족하면 커널이 자동으로 프로세스를 선택해서 죽인다.

```
# /var/log/syslog 또는 dmesg에서 확인 가능
Out of memory: Kill process 12345 (python3) score 900 or sacrifice child
Killed process 12345 (python3) total-vm:2048000kB, anon-rss:1024000kB
```

#### 이 과제의 MemoryGuard

`agent-leak-app`은 OS의 OOM Killer보다 먼저 스스로를 종료하는 내부 보호 정책(MemoryGuard)을 갖고 있다.

```
[CRITICAL] [MemoryGuard] Memory limit exceeded (256MB >= 256MB)
[CRITICAL] [MemoryGuard] Self-terminating process 12345
>>> [SYSTEM] SELF-TERMINATED (Memory Limit Exceeded) <<<
```

### 5.4 MEMORY_LIMIT 환경변수

이 과제에서 `MEMORY_LIMIT` 환경변수는 MemoryGuard의 임계치를 설정한다.

```
MEMORY_LIMIT=100  → 메모리가 100MB 초과하면 종료 (빨리 종료됨)
MEMORY_LIMIT=512  → 메모리가 512MB 초과해야 종료 (더 오래 생존)
```

**Before & After 실험의 핵심**: 이 값을 바꾸면서 프로세스 생존 시간이 달라지는 것을 관찰한다.

### 5.5 메모리 관련 역사

| 연도 | 사건 |
|---|---|
| **1965** | 가상 메모리 개념 도입 (Atlas 컴퓨터) |
| **1979** | C 언어의 malloc/free — 개발자가 직접 메모리 관리 |
| **1995** | Java의 GC(Garbage Collector) — 자동 메모리 관리 시작 |
| **2000년대** | 대규모 서버 운영에서 Memory Leak이 주요 장애 원인으로 부각 |
| **현재** | Rust 언어 — 컴파일 타임에 메모리 안전성 보장 (GC 없이) |

## 6. CPU 스케줄링과 과점유

### 6.1 CPU란?

CPU(Central Processing Unit)는 컴퓨터의 **두뇌**다. 모든 연산(더하기, 조건 판단, 메모리 읽기 등)을 수행한다.

현대 CPU는 여러 **코어(Core)** 를 가지지만, 동시에 실행 가능한 프로세스 수는 제한적이다.

### 6.2 CPU 스케줄링이란?

여러 프로세스가 동시에 CPU를 원할 때, OS가 **누구에게 CPU를 줄지** 결정하는 정책.

```
프로세스 A, B, C, D가 모두 CPU를 원함
        ↓
OS 스케줄러가 순서 결정
        ↓
A 실행 → B 실행 → C 실행 → A 실행 → ...
(아주 빠르게 전환되어 마치 동시에 실행되는 것처럼 보임)
```

이렇게 빠르게 전환되는 것을 **컨텍스트 스위칭(Context Switching)** 이라고 한다.

### 6.3 CPU 과점유란?

하나의 프로세스가 CPU를 지나치게 오랫동안, 또는 과도하게 사용하는 상태.

```
정상 상태:
CPU ─ A(10%) ─ B(20%) ─ C(5%) ─ D(15%) ─ 유휴(50%)

CPU 과점유:
CPU ─ agent-leak-app(95%) ─ 나머지(5%)
→ 다른 프로세스들이 CPU를 받지 못해 전체 시스템이 느려짐
```

### 6.4 CPU 과점유가 시스템에 미치는 영향

```
특정 프로세스 CPU 사용률 급상승
    ↓
다른 프로세스들 CPU 시간 부족
    ↓
웹 서버: 응답 지연 / DB: 쿼리 지연 / 전체 서비스 느려짐
    ↓
사용자 불만 / 타임아웃 에러 증가
```

### 6.5 이 과제의 Watchdog

`agent-leak-app`의 **Watchdog** 은 CPU 사용률이 `CPU_MAX_OCCUPY`를 초과하면 프로세스를 강제 종료한다.

```
[WARNING] [Watchdog] CPU usage 87% exceeded threshold 80%
[CRITICAL] [Watchdog] INITIATING EMERGENCY ABORT (SIGTERM)
>>> [SYSTEM] WATCHDOG: PROCESS TERMINATED <<<
```

### 6.6 CPU_MAX_OCCUPY 환경변수

```
CPU_MAX_OCCUPY=30  → CPU가 30% 초과하면 종료 (쉽게 종료됨)
CPU_MAX_OCCUPY=90  → CPU가 90% 초과해야 종료 (더 오래 생존)
```

### 6.7 CPU 관련 역사

| 연도 | 사건 |
|---|---|
| **1960년대** | 최초의 CPU 스케줄링 알고리즘 연구 (IBM OS/360) |
| **1970년대** | UNIX에서 Round-Robin 스케줄링 표준화 |
| **1990년대** | 멀티코어 CPU 등장 준비 — 병렬 처리 연구 활발 |
| **2000년대** | 멀티코어 시대 — 스레드 프로그래밍 보편화 |
| **현재** | CPU 과점유 탐지가 APM(Application Performance Monitoring)의 핵심 지표 |

## 7. Deadlock (교착상태)

### 7.1 Deadlock이란?

두 개 이상의 프로세스(또는 스레드)가 **서로 상대방이 가진 자원을 기다리며 영원히 멈추는** 상태.

### 7.2 식사하는 철학자 문제 (Dining Philosophers Problem)

Dijkstra(1965)가 제안한 고전적 비유 모델.

```
         [철학자 A]
        /           \
  [젓가락 5]    [젓가락 1]
      |                 |
[철학자 E]          [철학자 B]
      |                 |
  [젓가락 4]    [젓가락 2]
        \           /
         [철학자 C]
              |
         [젓가락 3]
              |
         [철학자 D]
```

**규칙**: 철학자는 밥을 먹으려면 양쪽 젓가락 두 개가 모두 필요하다.

**Deadlock 발생 시나리오**:
1. 모든 철학자가 동시에 왼쪽 젓가락을 집음
2. 이제 모두 오른쪽 젓가락을 기다림
3. 하지만 오른쪽 젓가락은 옆 철학자가 이미 집음
4. **→ 아무도 밥을 먹지 못하고 영원히 기다림**

### 7.3 Deadlock 4대 조건

Deadlock이 발생하려면 **4가지 조건이 모두 동시에** 충족되어야 한다. 하나라도 깨면 Deadlock은 발생하지 않는다.

#### 1. 상호 배제 (Mutual Exclusion)

자원은 한 번에 하나의 프로세스만 사용할 수 있다.

```
젓가락은 한 명만 사용 가능
→ 락(Lock)으로 구현됨
```

#### 2. 점유 대기 (Hold and Wait)

자원을 가진 상태에서 다른 자원을 기다린다.

```
철학자 A: 왼쪽 젓가락 쥔 채로(점유) 오른쪽 젓가락 기다림(대기)
```

#### 3. 비선점 (No Preemption)

자원을 강제로 빼앗을 수 없다. 자원 보유자가 스스로 놓아야 한다.

```
철학자 B의 젓가락을 강제로 빼앗을 수 없음
→ B가 자발적으로 내려놓을 때까지 기다려야 함
```

#### 4. 순환 대기 (Circular Wait)

프로세스들이 원형으로 서로를 기다린다.

```
A → B의 자원 기다림
B → C의 자원 기다림
C → A의 자원 기다림
→ 원형 대기 형성
```

### 7.4 컴퓨터에서의 Deadlock

```python
# 스레드 A
lock_X.acquire()          # X 잠금
time.sleep(0.1)           # 잠시 대기
lock_Y.acquire()          # Y 잠금 시도 → B가 Y를 가지고 있어 대기

# 스레드 B
lock_Y.acquire()          # Y 잠금
time.sleep(0.1)           # 잠시 대기
lock_X.acquire()          # X 잠금 시도 → A가 X를 가지고 있어 대기

# 결과: A는 Y를 기다리고, B는 X를 기다리며 둘 다 영원히 멈춤
```

### 7.5 Deadlock의 특징 — 왜 발견하기 어려운가?

| 특징 | 설명 |
|---|---|
| **프로세스가 살아있음** | 메모리에 존재하고, PID도 확인됨 |
| **CPU 사용률 0** | 아무것도 안 하므로 CPU를 쓰지 않음 |
| **로그가 멈춤** | 마지막 로그 이후 더 이상 출력 없음 |
| **응답 없음** | 요청을 보내도 응답이 돌아오지 않음 |

**→ "살아있는 좀비 프로세스"**

### 7.6 Deadlock 진단 방법

```bash
# 1. 프로세스가 살아있는지 확인
ps -ef | grep agent-leak-app

# 2. 스레드별 CPU/메모리 상태 확인
top -H -p <PID>
ps -L -p <PID>

# 3. 로그 파일 마지막 부분 확인
tail -n 50 <로그파일>
```

**Deadlock 징후**:
- PID는 존재
- CPU: 0.0%
- 로그: "WAITING..." 또는 "BLOCKED..." 이후 멈춤

### 7.7 MULTI_THREAD_ENABLE 환경변수

```
MULTI_THREAD_ENABLE=true  → 멀티스레드 활성화 → Deadlock 발생 가능
MULTI_THREAD_ENABLE=false → 단일스레드로 실행  → Deadlock 발생 안 함
```

### 7.8 Deadlock 해결 전략

| 전략 | 설명 | 단점 |
|---|---|---|
| **예방(Prevention)** | 4대 조건 중 하나를 원천 차단 | 자원 낭비, 복잡한 설계 |
| **회피(Avoidance)** | 자원 요청 전 안전 여부 계산 (Banker's Algorithm) | 성능 오버헤드 |
| **탐지(Detection)** | Deadlock 발생 후 탐지하고 해소 | 장애 발생 후 복구 필요 |
| **무시(Ignore)** | 발생 확률이 낮으면 그냥 재시작 | 데이터 손실 위험 |

### 7.9 Deadlock 역사

| 연도 | 사건 |
|---|---|
| **1965** | Dijkstra가 교착상태 개념 공식화 |
| **1965** | 식사하는 철학자 문제 제안 |
| **1971** | Banker's Algorithm 발표 (안전한 자원 할당 알고리즘) |
| **1980년대** | DB 트랜잭션에서 Deadlock 탐지/복구 표준화 |
| **현재** | 분산 시스템에서의 분산 Deadlock이 새로운 도전 |

```bash
dave  12345  1234  1 14:00 pts/0  00:00:05 python3 agent-leak-app

```

```bash

```

| 시그널 | 번호 | 의미 |
|---|---|---|
| SIGTERM | 15 | 정상 종료 요청 (프로세스가 처리 가능) |
| SIGKILL | 9 | 강제 종료 (OS가 직접 처리, 무시 불가) |
| SIGINT | 2 | Ctrl+C와 동일 |

## 10. 보너스: CPU 스케줄링 알고리즘

### 10.1 FCFS (First Come First Served)

**먼저 온 작업을 끝까지 처리한 후 다음 작업 처리**

```
도착 순서: A(10초) → B(2초) → C(5초)
실행: AAAAAAAAAA BB CCCCC
```

| 장점 | 단점 |
|---|---|
| 구현 단순 | 짧은 작업이 긴 작업 뒤에 오면 오래 기다림 (호위 효과) |

**적합한 서비스**: 배치 처리 시스템, 파일 인코딩 작업 등

### 10.2 Round-Robin (RR)

**각 프로세스에 동일한 시간(time quantum)을 순서대로 할당**

```
time quantum = 2초
도착: A(10초), B(2초), C(5초)
실행: AA BB CC AA CC AA AA AA → 모두 공평하게 처리
```

| 장점 | 단점 |
|---|---|
| 공평한 CPU 배분 | 컨텍스트 스위칭 오버헤드 |
| 응답 시간 단축 | time quantum 설정이 중요 |

**적합한 서비스**: 웹 서버, 대화형 시스템, 사용자 응답이 중요한 서비스

### 10.3 Priority Scheduling (우선순위)

**우선순위가 높은 작업을 먼저 처리**

```
A(우선순위: 3), B(우선순위: 1), C(우선순위: 2)
실행: BBBB...CCCCCC...AAAA...
```

| 장점 | 단점 |
|---|---|
| 중요 작업 빠른 처리 | 우선순위 낮은 작업이 무한 대기 (기아 현상) |

**적합한 서비스**: OS 커널 작업, 실시간 시스템

### 10.4 로그로 스케줄링 알고리즘 추론하는 법

로그에서 스레드의 실행 패턴을 본다.

```
[14:00:00.100] [Thread-A] Calculating... (10%)
[14:00:00.150] [Thread-A] Calculating... (20%)
[14:00:00.200] [Thread-B] Calculating... (10%)  ← A 중단, B 시작
[14:00:00.250] [Thread-B] Calculating... (20%)
[14:00:00.300] [Thread-C] Calculating... (10%)  ← B 중단, C 시작
[14:00:00.350] [Thread-A] Resumed. (30%)        ← C 중단, A 재개
```

분석:
- A가 100% 완료 전에 B가 시작됨 → FCFS 아님
- 특정 스레드가 독점하지 않음 → Priority 아님
- A→B→C→A 순환 패턴, 일정 간격 교체 → **Round-Robin 추론**

## 11. 개념 연결 요약

```
운영체제(OS)
├── 프로세스 관리
│   ├── 프로세스 상태 (Ready/Running/Waiting/Terminated)
│   ├── 스레드 (프로세스 내 실행 단위)
│   └── 컨텍스트 스위칭
│
├── 메모리 관리
│   ├── Heap / Stack / Code 영역
│   ├── Memory Leak (Heap 미해제)
│   └── OOM → MemoryGuard → SELF-TERMINATED
│
├── CPU 스케줄링
│   ├── Round-Robin / FCFS / Priority
│   ├── CPU 과점유 (한 프로세스가 독점)
│   └── Watchdog → SIGTERM
│
└── 동기화 / 자원 관리
    ├── Lock / Mutex
    ├── Deadlock 4대 조건
    │   ├── 상호 배제
    │   ├── 점유 대기
    │   ├── 비선점
    │   └── 순환 대기
    └── 식사하는 철학자 문제
```
